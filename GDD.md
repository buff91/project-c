## 1. 한 줄 콘셉트 (Elevator Pitch)

> 시야에 따라 층이 드러나는 아이소메트릭 던전에서, 적을 **구멍에 빠뜨리고 바닥을 무너뜨려** 층 아래로 떨어뜨리는 턴제 로그라이크.

## 2. 핵심 기둥 (Design Pillars)

게임의 모든 결정은 이 기둥들을 기준으로 판단한다. (기둥은 적게 유지 — 아래 4개가 상한)

1. **입체 공간(Verticality & Elevation)** — 재미는 "높이 차"에서 나온다.
   - 층(floor) 간 이동뿐 아니라 **한 층 안에서도 높이차(층고)** 가 존재. 계단으로 같은 층 내에서 오르내림 (Baldur's Gate 3식 멀티 높이 개념).
   - **낙하는 이 공간성을 활용하는 여러 상호작용 중 하나**일 뿐 — 핵심은 "지형/오브젝트/층과 입체적으로 상호작용하는 맛".
2. **상호작용 & 상태이상(Interaction & Status)** — 전투의 깊이는 여기서 나온다.
   - 지형·오브젝트·적과 상호작용 (오브젝트 밀기/부수기/작동).
   - 상태이상: **폭발 / 빙결(Freeze) / 화상(Burn)** 등. 요소끼리 **반응**(불+기름=확산, 물+빙결, 전기 등).
3. **제한된 시야(Limited Sight)** — 모든 걸 보여주지 않는다. 정보의 부재가 곧 긴장.
4. **파밍 & 조합(Farming & Crafting)** — 판을 진행하며 자원을 모으고 아이템을 조합. 메타 프로그레션과 연결.

> 창발적 전술(Emergent Tactics)은 위 기둥들이 충돌해 저절로 생기는 결과 — 목표이자 검증 기준.

## 3. 타깃 & 범위

- **타깃 플레이어**: 로그라이크/픽셀아트 팬, 짧게 여러 판 즐기는 모바일 유저
- **1차 목표**: 학습 + 완성 가능한 세로형 프로토타입 (출시는 그 다음)
- **개발 인원**: 1인 (개인 개발)
- **범위 원칙**: **잔인할 정도로 좁게.** 첫 목표는 "한 층에서 이동·전투·아이템이 도는 것".

## 4. 게임플레이 개요

### 4.1 기본 루프

```
타이틀 → 허브(준비/원정지 선택) → 던전 진입 → 층 탐험(FOV로 조금씩 드러남) → 전투(일반 + 층간 낙하)
   → 아이템/자원 획득 → 계단/구멍으로 다음 층 → 반복 → 사망 시 처음부터
```

- 타이틀과 허브는 독립 씬으로 유지한다. 이후 프롤로그·세계관·스토리 연출은
  `타이틀 → (스토리 씬) → 허브` 사이에 삽입하며 허브 월드나 던전 씬에 종속시키지 않는다.

### 4.2 조작 (모바일)

- **빈 타일 탭**: 목표 타일 이동 (A* 경로)
- **겹친 층 탭**: 화면에 실제 렌더된 현재 활성 층 타일을 우선한다. 실제 개구부로
  보이는 인접 층 조각은 현재 층 다이아몬드와 겹치지 않을 때만 선택할 수 있다.
- **적 탭**:
  - 근접 무기 → **인접 칸까지 이동 후** 공격 (붙어야 공격)
  - 원거리 → 조준 후 발사
- **오브젝트 탭**: 오브젝트 선택 시 **상호작용 버튼(들)이 팝업** (밀기 / 부수기 / 열기 / 줍기 등 문맥별)
- **높이 조준**: 다른 층·높이 대상 선택 시 타겟 마커 + 층/높이 표시
- 턴제이므로 실시간 반응속도 부담 없음 → 모바일에 적합
- ⚠️ **PC/모바일 동시 지원 고려** → 입력 추상화 필요 (아래 §12 참고)

## 5. 핵심 시스템

### 5.1 다층 격자 (Foundation)

- 논리 좌표: `GridPos { int x, y, int elevation }`
  - `elevation` = **연속적인 높이값**. 단순 "층 번호"가 아니라 **한 층 안에서도 높이차가 존재** (계단·경사로). Baldur's Gate 3처럼 하나의 층이 여러 높이 레벨을 가짐.
  - "층(floor)"은 elevation을 크게 묶은 논리적 구획 정도로 취급.
- 화면 좌표: 아이소 변환 + `elevation` 높이 반영 (정렬 우선순위도 elevation 기반)
- 같은 층 내 이동: 계단/경사로에서 elevation 변화 → 이동 비용·시야·조준에 반영
- **렌더링되는 높이 ≠ 시뮬레이션되는 범위** 를 분리 (성능, §5.7 참고)

### 5.2 시야 / 안개 (FOV)

- 타일 상태: `Unknown` / `Explored` / `Visible`
- 알고리즘: **Recursive Shadowcasting**
- 층 규칙:
  - 플레이어 층: 일반 FOV
  - 아래층: **구멍(Hole) 타일을 포털 삼아** 그 주변 반경만 Visible
  - 위층: 기본 차폐, 뚫린 곳 아래에서만 보임
- 수직 이동과 수직 시야는 분리한다. 같은 던전 층의 계단은 걸어서 통과하고, 사다리는 그 위에서 탭/상호작용해 elevation을 이동한다. `StairsUp/Down`은 밟는 즉시 활성 던전 층을 전환하며 다른 층을 투시시키지 않는다. 실제 `Hole`·개구부만 위·아래 양방향 시야 포털이다. 색은 보조 신호일 뿐이며, 계단=연속된 발판, 사다리=두 착지 높이에 딱 맞춘 세워진 레일, 층 전환=어두운 계단 아치+목적 층 표지, 개구부=깨진 바닥+아래 공간이라는 실루엣으로 먼저 구분한다.
- 적의 피해 숫자·사망 안내·적중 수는 `Visible`인 대상만 공개한다. 시야 밖 전투 결과는 시뮬레이션과 전적에는 반영하되 HUD 정보로 누설하지 않는다.
- 표현: tint + alpha (밝음 / 어둑 / 안 그림)

### 5.3 층간 전투 & 낙하 (시그니처 메카닉)

- 낙하 유발 트리거 (모두 `TryFall()` 하나로 수렴):
  - 넉백 공격으로 구멍으로 밀기
  - 약한 바닥(Weak Floor) 붕괴
  - 함정 발판 / 밀 수 있는 오브젝트
- 낙하 처리: 아래로 첫 단단한 바닥 탐색 → 낙하 층수 계산 → 낙뎀(누적 곡선) → 착지 지점 적 있으면 양쪽 데미지
- 플레이어도 동일 규칙 적용 (실수로 떨어질 수 있음 = 긴장감)

### 5.4 전투 (기본)

- 턴제: 플레이어 1행동 → 몬스터 1행동
- 적 시체는 결과 인지를 위해 기본 3턴간 남아 점점 흐려지고, 이후 월드 오브젝트·탭 대상·전투 목록에서 완전히 제거된다.
- 근접(**반드시 인접**) / 원거리(투사체 궤적) / 넉백
- HP, 간단한 스탯
- 낙하는 전투 요소 중 하나 — 아래층으로 **내려가는 수단으로도** 사용 가능 (계단 대신 뛰어내리기, 낙뎀 감수)

### 5.5 상태이상 & 요소 반응 (전투 깊이의 핵심)

- 상태이상: **화상(Burn)** · **빙결(Freeze)** · **폭발(Explosion)** · (확장: 감전/중독/기름/젖음 등)
- **요소 반응 규칙** — 시스템끼리 반응해 창발적 전술 생성:
  - 불 + 기름/가스 → 확산·폭발
  - 물/젖음 + 빙결 → 광역 결빙, 이동 봉쇄
  - 젖음 + 감전 → 광역 데미지
  - 화상 + 폭발통 → 연쇄 폭발
- 구현: 타일/엔티티에 **상태 플래그 + 지속턴**, 매 턴 반응 규칙을 평가하는 **반응 테이블**(데이터화)
- 낙하·오브젝트와 결합 → "적을 얼려서 밀어 떨어뜨리기" 같은 콤보

### 5.6 파밍 & 조합 (Crafting)

- 던전 진행 중 자원/재료 수집(파밍)
- 재료 조합으로 소모품/장비 제작 (예: 기름+병=화염병)
- 메타 프로그레션과 연결 (다음 판으로 이어지는 레시피 해금 등)
- **백팩은 6×4 멀티슬롯 격자**다. 소형 1×1, 긴 병·단검·두루마리 1×2,
  대형 유물 2×2처럼 아이템 하나마다 실제 면적을 점유한다.
- 모바일 프로토타입은 큰 아이템 우선의 결정론적 자동 정리를 사용한다. 공간이 없으면
  월드 아이템을 획득하지 않고 바닥에 남기며, 출정 백팩에 넣지 못한 물품도 창고에 둔다.
- 백팩의 같은 종류도 인스턴스별 면적을 차지한다. 허브 창고는 대량 보관용이므로 종류별 중첩을 유지한다.
- **허브에서 원정 물품을 직접 준비한다.** 창고와 6×4 출정 백팩을 나란히 보여 주며,
  모바일은 선택 후 반대편 빈 공간 탭, PC는 버튼 또는 드래그로 한 개씩 옮긴다.
- 선택한 영웅의 기본 지급품은 출정 백팩에 잠금 상태로 먼저 배치한다. 영웅 변경으로 공간이
  부족해지면 선택 물품을 삭제하지 않고 창고로 되돌린다.
- 던전에는 기본 지급품과 출정 백팩에 넣은 물품만 반입한다. 진입 시 선택 물품은 창고 메타에서
  원정 인벤토리로 이동하며, 선택하지 않은 창고 물품은 다음 원정을 위해 그대로 보존한다.

### 5.7 몬스터 AI & 활성화/컬링 (성능 핵심) ⭐

> FOV만큼이나 중요. **모든 몬스터를 매 턴 연산하면 안 된다.**

- **활성 반경(Activation Radius)**: 플레이어 주변 일정 범위 + 같은 방/시야 내 몬스터만 "활성" → 매 턴 사고
- 비활성 몬스터: 정지 또는 저비용 "휴면" 상태 (간헐 갱신)
- 다층: 보이는 층이라도 **구멍 근처 등 상호작용 가능한 몬스터만** 활성
- **몬스터 생성(spawn) 로직**: 절차생성 시 배치 규칙 + 진행 중 추가 스폰 규칙 (난이도·깊이 연동)
- 행동 결정: 단순 상태머신(순찰 → 추격 → 공격 → 도주)부터 시작, 필요 시 유틸리티 AI로 확장

### 5.8 아이템 / 던전 생성

- 무기, 소모품(폭탄=바닥 파괴 등), 열쇠, 조합 재료
- 절차적 던전 생성 (룸-앤-코리더 또는 BSP) + **높이차(계단/경사) 배치**

### 5.6 아트 파이프라인 (교체 자유)

- 로직 ↔ 비주얼 분리
- **ScriptableObject** 로 "타입 → 스프라이트/애니" 매핑
- 네이밍 규칙 + Sprite Atlas
- 픽셀아트 임포트 **프리셋** (Point filter, No compression, PPU 통일, Pixel Perfect Camera)

## 6. 아트 & 연출 방향

- 픽셀아트, 아이소메트릭 스프라이트 — **4방향 기본, 8방향도 적극 검토** (입체감·조준 표현에 유리, 아트 비용과 트레이드오프)
- **제작 파이프라인**: AI로 에셋 초안 생성 → **Aseprite로 다듬기 + 애니메이션 프레임 처리** → Unity 임포트 (프리셋 적용). Aseprite `.ase` → 스프라이트시트/JSON 워크플로 정립 필요.
- 낙하 연출: 떨어지는 애니메이션 + 카메라 살짝 아래 팬 + 흔들림
- 높이 조준 연출: 위/아래 보는 포즈 + 투사체 포물선 + 타겟 마커

## 10. 확정된 방향 (v0.2)

- **한 판 목표**: **최심층 도달** — 계속 아래로 내려가 가장 깊은 층/보스에 도달. 수직성 기둥과 완벽히 일치.
- **진행/죽음**: **메타 프로그레션** — 죽으면 그 판은 끝(로그라이트)이지만 일부 해금·성장이 다음 판으로 이어짐. 재도전 동기 확보.
- **테마**: **정통 판타지 던전** — 고블린/해골/슬라임/마법. 에셋 확보 쉽고 픽셀아트 레퍼런스 풍부. 안전한 출발.

### 이 결정들이 파생시키는 것 (추후 설계 항목)

- 메타 프로그레션 저장 데이터: 무엇을 해금? (직업? 무기? 시작 아이템? 스탯?)
- "최심층" = 몇 층? 한 판 목표 길이 (예: 20~30분)
- 보스/최심층 보상 연출
- 정통 판타지 → 몬스터/아이템 로스터 초안 필요

## 11. 아직 안 정한 것 (Open Questions)

주석에서 "다루자"로 표시됨 → 아래에 초안 방향 제안 (확정은 M1 이후 실플레이로):

- **메타 프로그레션 해금 대상** → 제안: ① 조합 레시피, ② 시작 장비/아이템, ③ (직업제 채택 시) 직업 해금. 영구 스탯 강화는 로그라이트 밸런스를 무너뜨리기 쉬워 신중.
- **최심층 깊이 / 한 판 길이** → 제안: 프로토는 5~8층, 한 판 20~30분 목표. 완성본에서 확장.
- **단일 영웅 vs 직업 선택** → 제안: 프로토는 **단일 영웅**으로 단순화, 시스템 검증 후 직업제 도입 검토.
- **층 개수 / 던전 규모** → 위 최심층 깊이와 함께 M1~M3에서 실측 조정.

## 12. PC / 모바일 크로스플랫폼 고려사항 (신규)

> 둘 다 지원하려면 처음부터 **입력과 UI를 추상화**해야 나중에 안 갈아엎는다.

- **입력 추상화**: "탭 = 클릭", "롱프레스 = 우클릭/호버" 등을 게임 로직이 아닌 **입력 레이어**에서 통일. Unity **Input System(new)** 사용 → 터치/마우스/키보드/게임패드를 액션 단위로 매핑.
- **호버 부재 문제**: PC는 마우스 호버로 정보 툴팁을 주지만 모바일엔 호버가 없음 → **탭=선택, 재탭=실행** 2단계 또는 롱프레스로 정보 표시.
- **UI 스케일/해상도**: Canvas Scaler로 다양한 화면비(모바일 세로/가로, PC 와이드) 대응. 안전영역(노치) 처리.
- **정밀 조준**: 마우스는 픽셀 단위, 터치는 손가락이 굵음 → **타일 단위 스냅 + 확인 단계**로 두 입력 모두 커버 (턴제라 유리).
- **성능 예산**: 모바일이 하한선 → 몬스터 컬링(§5.7)·렌더 층 제한이 여기서도 중요.
- **저장/동기화**: 추후 크로스 세이브 고려 시 저장 포맷을 플랫폼 독립적으로.
