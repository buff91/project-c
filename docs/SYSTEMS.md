# 시스템 설계 요약 (에이전트용)

> `GDD.md` §5의 상세를 코딩 판단용으로 압축. 충돌·모호 시 **GDD가 최종 출처**.

## 백팩 / 창고 — §5.6

- 던전 백팩은 `BackpackRules`의 6×4 격자다. `ItemKind → ItemFootprint` 매핑과
  큰 아이템 우선 자동 배치를 Core에서 계산하고 UI는 `BackpackLayout`을 그대로 그린다.
- 1×1 소형, 1×2 기름 병/투척 단검/귀환 두루마리, 2×2 유물로 시작한다. 회전은 없다.
- 백팩 공간 부족 시 획득은 실패하고 월드 아이템은 활성 상태로 남는다.
- 허브 창고는 48칸 표현을 쓰되 종류별 중첩 저장이며, 던전 백팩의 개별 면적 규칙과 구분한다.
- `ExpeditionLoadoutRules`가 창고 ↔ 출정 백팩의 한 개 단위 이동, 영웅 기본 지급품 포함 배치,
  공간 부족 거부와 영웅 변경 시 초과분 창고 복귀를 담당한다.
- `MetaSaveData`는 창고 수량과 출정 선택 수량을 별도로 저장한다. 허브 UI는 모바일 선택+반대편 탭과
  PC 드래그/버튼을 같은 전송 규칙으로 호출하고, 실패한 이동은 데이터 변경 없이 빨간 드롭 피드백을 낸다.
- 던전 진입 시 `ConsumeLoadout()`이 선택 물품만 런 인벤토리로 이동한다. 선택하지 않은 창고 물품은
  유지하며, 예외적인 배치 초과분도 삭제하지 않고 창고로 되돌린다.

## 다층 격자 (Foundation) — §5.1

- 논리 좌표 `GridPos { int x, y, int elevation }`.
  - `elevation` = **연속 높이값** (단순 층 번호 아님). 한 층 안에서도 높이차 존재 (계단/경사, BG3식 멀티 높이).
- "층(floor)" = elevation을 크게 묶은 논리 구획.
- 구현 규칙: `DungeonHeightModel`이 **elevation 4칸을 던전 한 층**으로 묶는다.
  - B1(floor 0) = e0..e3, B2(floor -1) = e-4..e-1, B3(floor -2) = e-8..e-5.
  - 음수 elevation도 floor/local height 왕복 변환이 정확해야 한다.
- 같은 층의 높이 변화는 인접 `Stairs`를 통해 ±1만 허용한다.
- 던전 층 이동은 `GridMap.Connect()`의 명시적 양방향 링크로 `StairsUp ↔ StairsDown`을 연결한다.
- 시뮬레이션은 플레이어가 있는 한 층만 활성화한다.
- 던전 표시는 두 모드로 분리한다.
  - `PLAY FOV`: 현재 층은 `Visible / Explored / Unknown`. 닫힌 문은 문 자체까지만 보이고 뒤쪽 전파를 막는다.
  - `DEBUG ALL`: 모든 층·방·문을 표시하고 층 사이를 작은 간격으로 벌려 생성 구조와 정렬을 검사한다.
- PLAY에서 다른 층의 `Explored` 기록은 미니맵/세이브에는 유지하지만 월드 지오메트리에는 표시하지 않는다. 실제 `Hole`·개구부를 현재 위치에서 볼 수 있을 때만 반대편 국소 범위를 반투명 표시한다.
- 수직 이동은 네 종류로 분리한다: `Stairs`는 같은 던전 층 안에서 걸어서 통과하는 elevation 이동, `Ladder`는 같은 층의 두 높이를 명시적 링크로 잇고 그 위에서 탭/Space로 사용, `StairsUp/Down`은 진입 즉시 반대편 링크까지 한 행동으로 소비하는 활성 던전 층 전환, `Hole`은 낙하와 양방향 시야가 가능한 실제 개구부다.
- 사다리 월드 오브젝트는 실제 elevation 차이와 타일 높이로 길이를 계산해 두 착지면 사이에만 걸친다. 디버그 전체 보기에서도 활성 층의 현재 도달 가능한 랜드마크만 표시하고, 층 전환 링크는 개구부처럼 샤프트로 그리지 않는다.
- `StairsUp/Down`은 PLAY에서 다른 층을 노출하거나 점선 샤프트를 그리지 않는다. 전체 링크 구조는 `DEBUG ALL`에서만 표시한다.
- 발견된 `Hole`은 시작/착지 다이아몬드와 청록 점선 샤프트로 연결한다. 위에서는 착지점 주변을 내려다보고, 아래에서는 같은 개구부 주변을 올려다볼 수 있다.
- PLAY의 개구부 반대편 조각은 반경 2칸을 현재 층보다 위/아래로 조금 더 벌린 컷어웨이로 표시한다. 전체 층은 노출하지 않는다.
- 수직 수단은 색 범례를 외우게 하지 않는다. 계단은 단차 발판+`WALK`, 사다리는 세워진 레일+`CLIMB`, 층 전환은 어두운 아치+`Bn ▲/▼`, 개구부는 깨진 테두리+반대편 층 표지로 월드에서 직접 읽힌다.
- 해당 종류가 시야에 처음 들어온 순간 `VerticalRouteCue`를 1회 발생시켜 7초짜리 비차단 발견 카드를 보여준다. 카드 문장은 생김새·조작·목적지를 한 문장으로 연결하며, 플레이어가 사다리/층 전환 위에 서면 하단 HUD도 현재 행동을 우선 표시한다.
- 화면 좌표 = 아이소 변환 + elevation 높이 반영. **정렬 우선순위도 elevation 기반** (`IsoGrid`에 집중).
- 시점 회전은 카메라 Transform을 돌리지 않고 `IsoGrid`가 회전 중심 기준으로 `(x, y)`를 90도 단위 투영한다.
  - `GridToWorld`, `WorldToGrid`, `SortingOrder`가 같은 `viewQuarterTurns(0..3)`를 사용한다.
  - 회전 시 타일/액터/소품을 다시 배치하고, 현재 앞쪽 두 변은 열어 두며 뒤쪽 두 변의 벽만 다시 만든다.
  - 2D 스프라이트를 한 장만 쓰는 프로토타입 방식이며, 최종 캐릭터는 같은 회전 이벤트로 4방향 스프라이트를 선택한다.
- **렌더링 높이 ≠ 시뮬레이션 범위** 분리 (성능).

## 시야 / 안개 (FOV) — §5.2

- 타일 상태: `Unknown` / `Explored` / `Visible`.
- 알고리즘: **Recursive Shadowcasting (구현됨, `GridVisibility`)** — (x,y) 2D 8옥탄트 캐스팅 + 컬럼별 elevation 대역 표면 스캔. 반경은 체비셰프.
  - **void(타일 없는 컬럼) = 불투명.** 이 던전은 방 경계를 벽 타일이 아니라 타일 부재로 표현하므로(벽은 비주얼 전용), 이 규칙이 "닫힌 문 뒤 방은 Unknown" 불변식의 핵심.
  - 닫힌 문/벽은 그 칸까지만 보이고 너머 차단. **Hole/약한 바닥/계단은 투과** (`HasLineOfSight`와 동일 기준 — 원거리/폭탄과 시야가 일치).
  - 1단 높이차(raised)는 시야를 막지 않는다. 필요 시 표면 판정에 눈높이 임계 규칙을 추가.
- 층 규칙: 플레이어 층 일반 FOV / 아래층은 **구멍(Hole)을 포털 삼아** 반경만 Visible / 위층은 뚫린 곳 아래에서만.
- 표현: tint + alpha.
- `PLAY FOV`의 미탐색 영역은 실제 방·복도 모양 대신 층 생성 범위 전체를 덮는 어두운 아이소 안개 배경으로 표시한다. 카메라 void와 색·외곽선을 달리해 플레이 영역은 읽히되, 아직 발견하지 않은 구조는 노출하지 않는다. 이 배경은 입력 대상이 아니며 실제 이동 가능 여부는 항상 `GridMap.Has()`가 결정한다.
- 적 전투 피드백도 같은 FOV를 따른다. 시야 밖 적의 피해 플로팅·플래시·사망 문구는 숨기고, 폭발 적중 수는 보이는 대상만 센다. 전투 결과와 처치 전적 자체는 그대로 반영한다 (`EnemyPresentationRules`).

## 층간 전투 & 낙하 (시그니처) — §5.3

- 모든 낙하 트리거는 `TryFall()` 하나로 수렴: 넉백→구멍, 약한 바닥 붕괴, 함정/밀 수 있는 오브젝트.
- 처리: 아래로 첫 단단한 바닥 탐색 → 낙하 층수 → 낙뎀(누적 곡선) → 착지점 적 있으면 양쪽 데미지.
- **플레이어에도 동일 규칙 적용** (실수 낙하 가능 = 긴장).
- 구현 (M4, `FallRules`/`KnockbackRules`): 낙뎀 곡선 = floors×(floors+1) (1층 2, 2층 6, 3층 12). 착지점에 산 점유자가 있으면 양쪽 낙뎀 후 낙하자는 인접 빈 칸으로 밀려난다. 넉백은 폭발 중심 반대 우세 축 1칸 — 벽/닫힌 문/점유는 무효, 구멍/허공이면 TryFall. 약한 바닥은 밟거나(이동) 충격(넉백 착지)으로 붕괴해 구멍이 된다. 폭발 반경 안의 폭발통은 유폭해 연쇄 폭발한다.

## 전투 (기본) — §5.4

- 턴제: 플레이어 1행동 → 몬스터 1행동.
- 근접(**반드시 인접**) / 원거리(투사체 궤적) / 넉백. HP + 간단 스탯.
- 검증 전투는 HUD에서 `MELEE / RANGED`를 전환한다. 근접은 적 인접 칸까지 자동 접근하고, 원거리는 같은 elevation·최대 사거리·Bresenham 시야선을 모두 만족해야 명중한다. 닫힌 문, 벽, 빈 타일은 사격선을 차단한다.
- 시체는 기본 3턴간 HP 바 없이 회색으로 남아 매 턴 흐려진다. 수명이 끝나면 씬 오브젝트와 적 목록에서 함께 제거되어 월드 탭 보정에도 더는 잡히지 않는다.
- **원거리는 근접보다 약하고 짧다** (기본 피해 1·사거리 4 vs 근접 2): 지각이 플레이어 FOV 대칭 + 이동속도 동일 구조라, 무비용 원거리가 근접과 같은 피해면 접근하는 몬스터를 전부 무피해 카이팅으로 잡는다(밸런스 시뮬로 확인). 원거리는 마무리/견제 수단.
- 낙하는 하강 수단으로도 사용 가능 (계단 대신 뛰어내리기).
- Hole을 탭하면 적 점유 칸을 피해 인접 칸으로 접근한 뒤 `FindLandingBelow()`의 첫 단단한 바닥으로 낙하한다. 현재 프로토타입은 이동만 처리하며 낙뎀·충돌 피해는 M4 `TryFall()`에 통합한다.

## 상태이상 & 요소 반응 — §5.5

- 상태이상: 화상(Burn) / 빙결(Freeze) / 폭발(Explosion) (확장: 감전/중독/기름/젖음).
- 반응 예: 불+기름=확산·폭발, 물/젖음+빙결=광역 결빙, 젖음+감전=광역뎀, 화상+폭발통=연쇄.
- 구현: 타일/엔티티에 **상태 플래그 + 지속턴**, 매 턴 평가하는 **반응 테이블(데이터화)**.
- 구현 (M4, `StatusEffects`/`StatusReactions`): 참가자별 상태 집합 — 화상=턴당 틱뎀, 빙결=턴 스킵, 재부여는 지속 연장. 반응 테이블 첫 항목은 불↔얼음 상쇄(해동/진화). 폭발 생존자는 화상 2턴. 턴 파이프라인은 **활성화 → 상태이상 틱 → Decide → 실행**이며 휴면(컬링) 중에는 틱도 멈춘다.

## 파밍 & 조합 — §5.6

- 진행 중 자원 수집 → 조합으로 소모품/장비 제작 (예: 기름+병=화염병) → 메타 프로그레션 연결.

## 몬스터 AI & 활성화/컬링 (성능 핵심) ⭐ — §5.7

- **활성 반경**: 플레이어 주변 + 같은 방/시야 내 몬스터만 "활성" → 매 턴 사고. 나머지는 휴면(간헐 갱신).
- 다층: 보이는 층이라도 구멍 근처 등 상호작용 가능한 몬스터만 활성.
- 스폰: 절차생성 배치 규칙 + 진행 중 추가 스폰 (난이도·깊이 연동).
- 행동: 단순 상태머신(순찰→추격→공격→도주)부터, 필요 시 유틸리티 AI로 확장.
- 구현 규칙 (M2, `MonsterBrain`/`MonsterActivation`/`MonsterArchetype`):
  - 파이프라인: **활성화 판정 → (M4: 상태이상 틱 자리) → Decide(의도 반환) → 실행**. 브레인은 Wait/Step/OpenDoor/Attack "의도"만 반환하고 위치·타일·연출 변경은 Gameplay가 실행 — M4 낙하(TryFall)·상태이상이 이 파이프라인에 끼어든다.
  - 지각 = **플레이어 FOV의 대칭**(`seenByPlayer` 콜백) + 어그로 반경(체비셰프). 같은 턴 다중 이동은 플레이어와 가까운 순으로 순차 처리해 점유 충돌을 없앤다.
  - 몬스터는 **추격 중에만 닫힌 문을 연다**(행동 1회). 순찰 중에는 열지 않으므로 "문 뒤 스폰" 구조가 유지된다. 시야를 잃으면 마지막 목격 지점까지 수색 후 순찰 복귀.
  - 활성화 = 같은 층 && 활성 반경(FOV+2). 규칙 목록 형태로 유지 — 층간 규칙("보이는 구멍 인접 아래층 몬스터")은 추후 추가.
- 구현 규칙 (M5, `MonsterRoster`): 몬스터 3종 — 고블린(공2·빈사 시 도주), 해골(단단·강타·비도주), 슬라임(약함·넓은 배회). 스탯·깊이 혼합표는 roster 한 곳에. **진행 중 추가 스폰**: 12턴 간격, 활성 층 몬스터 4마리 미만일 때 플레이어 시야 밖 스폰 지점에 깊이 혼합으로 1마리.
- **하행 계단 경비병**: 북쪽 방 스폰만으로는 완주 동선(남쪽 방→하행 계단)이 전투를 완전히 우회한다(시뮬로 발견). 하행 계단 인접(체비셰프 3)에 1+depth 마리를 추가 배치 — 깊을수록 무거운 관문. 하행 방은 도착 방과 항상 달라 "문 열기 전 차단" 스폰 불변식은 유지된다.

## 아이템 / 던전 생성 — §5.8

- 무기, 소모품(폭탄=바닥 파괴 등), 열쇠, 조합 재료.
- 구현된 소모품 규칙 (`Items.cs`): **물약** = HP 회복, 행동 1회 소비. **폭탄** = 사거리·시야선 확보된 단단한 타일에 투척, 같은 elevation 3×3 피해(던진 본인 포함 — 자폭 가능) + 화상 + 넉백, 빈 약한 바닥 붕괴·폭발통 유폭. **냉기 폭탄** = 낮은 피해 + 빙결 2턴, 폭발통은 유폭하지 않음(불이 아님). 낙하 처리는 `TryFall()`로 수렴.
- 절차 생성 (룸-앤-코리더 또는 BSP) + 높이차(계단/경사) 배치.
- 생성기는 층마다 **방 3개(+확률적 막다른 분기 방) → 1칸 복도 → 복도 목의 문** 그래프를 유지한 채, 방 크기/위치·복도·문·내부 계단·구멍·적 스폰을 **seed로 변형**한다. 같은 seed는 같은 던전을 재현한다.
- 생성 불변식 (EditMode 테스트로 고정): 문을 모두 열면 모든 walkable 타일이 입구에서 도달 가능하고, 문을 닫으면 북쪽 방이 차단된다. 구멍은 **정확히 한 층 아래의 walkable 칸**에 착지하며, 윗층 구멍의 착지 칸을 다시 뚫지 않는다(2층 관통 방지). 아래층 북쪽 방은 윗층과 기둥이 겹치게 제약해 구멍 후보가 항상 남는다.
- 스폰 규칙: **적 스폰은 문 뒤(북쪽 방)에만** 깊이 비례 1~4개 — 입구 방에 두면 진입 즉시 인접 전투가 강제되고 문 차단 불변식도 깨진다. **아이템**(물약/폭탄)은 북쪽·동쪽 방 빈 바닥에 1~2개, 막다른 분기 방에는 보상 1개를 보장한다. 적·아이템·입구·특수 타일은 서로 겹치지 않는다.
- `DoorClosed`는 단단한 바닥이지만 이동·시야를 막는다. 인접 상호작용으로 `DoorOpen`이 되면 이동·시야가 통과한다.
- PLAY 시야 계산도 같은 문 상태를 사용한다. 닫힌 문 뒤 방은 Unknown이고, 문을 연 직후 FOV를 다시 계산해 다음 공간을 드러낸다.
- 문 열기와 닫기는 플레이어 행동 1회를 소비한다. 자동 경로가 닫힌 문을 무시하고 통과해서는 안 되며, 상태 전환 때 문짝 접힘·상호작용 버스트·HUD 피드백을 함께 갱신한다.
- 문 비주얼은 정면 사각형을 사용하지 않는다. 통로가 남북이면 동서 평면, 통로가 동서면 남북 평면에 문틀을 세우고 `IsoGrid` 회전에 맞춰 사선 방향을 다시 고른다.
- 이후 확장(진짜 룸-앤-코리더/BSP)에서도 이 연결 그래프와 생성 불변식을 유지한다.

## 아트 파이프라인 — §5.6(art) / §6

- 로직↔비주얼 분리. **ScriptableObject로 "타입→스프라이트/애니" 매핑**.
- 픽셀아트 임포트 프리셋: Point filter, No compression, PPU 통일, Pixel Perfect Camera.
- 4방향 기본, 8방향 검토. 제작: AI 초안 → Aseprite 다듬기/애니 → Unity 임포트(프리셋).

## 크로스플랫폼 (PC/모바일) — §12

- **입력 추상화**: Unity Input System(new)으로 터치/마우스/키보드/게임패드를 액션 단위 매핑. 게임 로직에 플랫폼 분기 금지.
- **월드 탭 경계**: UI 차단 뒤 `GridMap.Has()`로 실제 타일인지 확인한 입력만 게임플레이에 전달한다. 화면의 검은 여백(void)은 미탐색 칸이 아니며, 이동·조준·자동 이동 취소를 일으키지 않는다. 벽은 맵 안의 타일이므로 전달한 뒤 이동 불가 피드백을 준다.
- **다층 월드 픽킹**: 모든 elevation 평면을 역산해 존재하는 타일을 고르지 않는다. 실제
  렌더된 아이소 다이아몬드의 `VisualPosition`으로 히트 테스트하고, 겹치면 현재 활성 층
  → 개구부 인접 층 미리보기 → 같은 레이어의 렌더 정렬 순서 순으로 선택한다.
- 호버 부재: 탭=선택, 재탭=실행 2단계 (또는 롱프레스로 정보).
- 조준: 타일 단위 스냅 + 확인 단계 (턴제라 유리).
- UI: 화면 고정 HUD는 UI Toolkit의 `PanelSettings(Scale With Screen Size)`로 화면비 대응, 안전영역(노치). 월드 앵커 UI만 UGUI를 사용한다. 저장 포맷은 플랫폼 독립적으로.
- 카메라: 던전은 플레이어 추적 구도를 유지한다. 고정 크기 허브는 투영된 맵 경계를 기준으로 중심과 직교 카메라 크기를 화면비별 계산해 모든 주요 지점을 노출한다.

## 프런트엔드 씬 흐름

- Build Settings 순서: `MainMenu` → `Hub` → `IsoPrototype`.
- `MainMenu`는 타이틀·이어하기·설정만 소유하고, `Hub`는 캠프 월드와 원정지 선택만 소유한다.
- 새 게임 경로는 `MainMenu → Hub → IsoPrototype`이다. 이후 세계관/프롤로그는
  `MainMenuController.StartNewGame()`의 다음 목적지를 별도 스토리 씬으로 바꿔 삽입한다.
- 던전 게임 메뉴의 `로비로 가기`는 체크포인트를 보존하고 `Hub`로, 게임오버의
  `메뉴로`는 `MainMenu`로 돌아간다.
